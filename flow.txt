「すべてをIT化が解決しない可能性」問題
「どの程度の量なら許容されるか」「他の疾患との兼ね合いでどう調整すべきか」といった複雑な個別要因までをシステムが自動で判断・加味することは難しく、結局は医師や薬剤師による「人」の確認と判断に頼らざるを得ない

ベテランの医療従事者が持っている「この患者さんには、この薬は相性が悪い気がする」という直感（暗黙知）は、本パーソン自身も明確に言語化できていないことが多いです。本人が言語化できないものを、ITエンジニアがシステム（形式知）に組み込むことは不可能
-------------------
GASスクリプトはﾒﾓとして保存　実際の実行はGAS画面での実行が必要
******************************************************************************
Pythonコードの編集状況の把握であれば、構文崩れ（インデントやクォーテーションの自動変換）を防ぐ観点で「メモ帳」の方が圧倒的に安全
（Googleドキュメントなどは「見やすさ」のために勝手にスペースを詰めたり、引用符（"や'）をスマートクォーテーション（特殊な記号）に変換してしまうため、コードがエラーになる原因を作ってしまいます。）
➡ｴﾃﾞｨﾀは構文修正機能もあり　コードを貼り付けた後、右クリック。 「ドキュメントのフォーマット (Format Document)」 をクリック。
------------------
python build_final.py
これでindex.htmlが生成されます。確認してよければ、
bash
git add .
git commit -m "コメント"
git push

もし書き換えを元に戻したくなった場合は、
bash
git restore template.html
で直前のコミット時点に戻せます。
（スタッシュが必要なのは「コミットしていない変更を一時退避して別ブランチに切り替えたい」ときで、また別の方法です。）
--------------------
git restore は、「まだコミットしていない書きかけのファイルを元に戻す」ための安全なピンポイント修正コマンドです。
	git restore main.py （main.py の書きかけの変更を取り消す）
	git restore . （今変更しているすべてのファイルの書きかけを取り消す）
git reset --hard は、「過去のセーブ地点まで、書きかけのコードも含めてすべてを強制的に巻き戻す」強力で少し危険なコマンドです。
	git reset --hard HEAD （現在の最新コミットの状態に完全に戻す。書きかけのコードは消滅します）
	git reset --hard [コミットID] （指定した過去のセーブ地点に完全に戻す
		Gitでコミット（セーブ）を実行すると、Gitが裏側で自動的に a1b2c3d... のような40桁の長い英数字のIDを発行します。これが「コミットID」（またはハッシュ値）

□□□履歴を確認するコマンド（git log）
① 1行でスッキリ見たいとき（超おすすめ）bashgit log --oneline
ここが凄い：1つのコミットが「短いコミットID ＋ メモ」だけで1行にまとまります。画面がスクロールだらけにならず、過去のセーブ地点一覧
② 最近の3つだけ見たいときbashgit log -n 3
ここが凄い：直近の3件（数字は変えられます）だけを表示して自動で終了します。画面がログで埋め尽くされないのでスマートです。
③ 誰がいつ変更したか、グラフィカルに見たいときbashgit log --oneline --graph --all
ここが凄い：ブランチ（枝分かれ）の状況が、文字のアート（線画）でツリー状に表示されます。チーム開発ではこれが必須になります。
